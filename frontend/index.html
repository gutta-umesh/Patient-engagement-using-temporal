Parent-Child Workflows

PatientJourneyWorkflow (parent) runs durable timers for 90 days. For each scheduled check-in, it executes a CheckInWorkflow (child), keeping histories short, clean, and isolated.

⏰

Durable Timers

Timer states are fully persisted in the Temporal database rather than keeping threads active. Timers resume flawlessly even if the cluster or workers crash.

📨

Durable Signals & Races

We employ a wait_condition signal race with a timer deadline. Whichever wakes the workflow first decides the outcome, avoiding stale states.

🔒

Activity Isolation

Workflows are completely pure and deterministic. Side effects (sending automated Emails, calling EHR FHIR REST APIs, scoring symptoms) are strictly contained inside isolated activities.

Activities implement robust exponential retries on failure.

🔄

Deterministic Replay

Workflows are replayed by Temporal to recreate the current execution state. We guarantee exact deterministic execution by avoiding random values or raw system times inside the workflow code.

🔁

Transactional Consistency

Patient monitoring logs are transactional. Failure to submit outcomes is handled automatically via Temporal's queue persistence, ensuring no patient's data is lost or un-logged.
